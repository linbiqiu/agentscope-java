# 飞书企业智能体服务 - 需求文档（MVP）

日期：2026-05-24  
状态：Updated

## 1. 目标与范围

构建一套给企业员工使用的飞书智能体服务，入口为飞书机器人私聊与群聊 @ 机器人。MVP 目标是先稳定跑通“员工可用、运营可控、技能可热更新”的最小闭环。

当前阶段明确范围：
- 员工只能使用，不能安装/修改技能。
- 运营通过配置为 agent 绑定可用技能。
- 技能仓库先使用指定目录（手工上传文件）。
- 技能变更通过运行时自动识别并加载（无需应用重新发版）。

当前阶段不做：
- 技能上传下载平台化界面。
- 版本化发布链路（v1/v2）与业务回滚接口。
- 复杂运维同步面板。

## 2. 角色与权限

### 2.1 角色
- 员工：仅通过飞书使用智能体。
- 运营管理：维护 agent 技能绑定并发布生效。
- 运维审计：查看调用与审计日志。

### 2.2 权限边界
- 员工端无技能管理入口。
- 仅运营端可变更绑定配置。
- 运行时只允许调用“已发布绑定且在技能仓库中存在”的技能。

## 3. 技能仓库与生效方式（MVP）

### 3.1 技能仓库
- 使用文件夹作为统一技能仓库。
- 每个技能目录包含 `SKILL.md`。
- 多个 agent 共享同一仓库技能，不做技能副本。

### 3.2 自动加载
- 应用运行时定时刷新技能目录。
- 新增/删除技能目录在刷新周期内自动生效。
- 不需要每次技能变更都重新发版。

## 4. 配置与部署约束

关键环境变量：
- `FEISHU_RUNTIME_SKILL_REPO_DIR`：技能仓库目录。
- `FEISHU_RUNTIME_SKILL_REPO_REFRESH_INTERVAL_MS`：刷新周期。

测试与生产均通过 env 配置注入，便于随时调整。

## 5. 运行时授权与编排规则

请求在运行时按以下规则处理：
1. 若携带 `requested_skill`，且该技能在“已发布绑定 + 仓库已加载技能”交集中，则优先执行该技能。
2. 若未指定或不可用，则由 AgentScope 原生编排在已加载技能中自动匹配最合适技能。
3. 若无匹配技能，才直接回答，并明确说明无匹配技能。

授权边界保持不变：
- 仅允许运行“已发布绑定且仓库存在”的技能。
- 员工无权增删改技能，仅可使用。

## 6. 会话与隔离

会话 key 继续按私聊/群聊隔离：
- 私聊：`agent:{agentId}:p2p:user:{userId}`
- 群聊：`agent:{agentId}:group:{conversationId}:user:{userId}`

## 7. 身份信息透传与用户档案约束（新增）

### 7.1 消息身份解析
- 发言人身份来自事件 `sender_id`。
- 被 @ 人身份优先读取结构化 `mentions[]`。
- 若 `mentions[]` 缺失但消息内容存在 `<at ...>...</at>`，需继续解析内容中的 at 标签作为补充来源。

### 7.2 标识语义
- 事件主用户标识 `userOpenId` 仅使用 `openId`，不再回退为 `unionId/userId`。
- `unionId/userId` 仅作为补充身份字段，通过身份上下文透传给运行时。

### 7.3 用户档案持久化
- 身份读取策略为 DB-first：`openId -> unionId -> userId`，未命中再调用飞书接口。
- 档案表写入采用 upsert 去重，不产生重复记录。
- 当用户信息不完整（缺少 `openId/mobile/employeeNo/email` 任一字段）时，不写入用户档案表。

### 7.4 运行时透传范围
- 发言人与被 @ 人身份必须通过 `identityContextJson` 透传到 runtime。
- runtime 需将身份上下文提供给 skill / tool / MCP，作为可消费的运行时上下文键值。

## 8. 验收标准（当前阶段）

- 员工无法安装或调整技能。
- 运营可通过草稿/发布控制 agent 可用技能。
- 请求带 `requested_skill` 且可用时，系统优先执行该技能。
- 未指定技能时，系统可在已加载技能中自动匹配并执行。
- 仓库新增/删除技能后，运行时可自动识别变更并生效。
- 不需要应用重新发版即可生效技能变更。
- 消息内 @ 人身份可由 `mentions[]` 与 `<at ...>` 双通道解析。
- `userOpenId` 仅为 openId。
- 不完整用户信息不写入用户档案表。
- skill/tool/MCP 可读取发言人和被 @ 人身份上下文。
